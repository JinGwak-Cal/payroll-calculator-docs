- **버전**: 06-디자인언어-벤치마킹실행 (검증 로그 + 확정 원칙 7-1~4 추가)
- **작성**: Claude (현업1-1 협업)
- **기준 시각**: 2606.13
- **목적**: ① 디자인 언어 4종 정의를 고정 ② Apple HIG·토스 골격 조사 결과를 4축 검증 근거로 보존

## 1. 디자인 언어(Design Language) 정의 — 고정 [4축 갱신]

> "디자인 언어 = 통일된 표현 규칙." 벤치마킹 시 아래 4개 질문을 체크리스트로 사용한다.
> (3축→4축 진화에 맞춰 **Type 언어 신설**. 2606.13)

| 언어 | 핵심 질문 | 축 | 1순위 검증 앱 |
|------|-----------|----|----|
| **Type 언어** | 이 앱은 UI의 *존재 목적*을 어떻게 분류하나? (Toast=단순알림 vs Feedback 독립체계 / Drawer=팝업 vs Overlay 독립계층) | Type | Apple HIG |
| **컴포넌트 언어** | 카드·버튼·칩을 한 규칙으로 통일했는가? | Component | 토스 |
| **기능 언어** | 기능마다 다른 UI를 한 톤으로 묶었는가? | Feature | 토스 |
| **레이어 언어** | 입력·결과·오버레이를 어떤 위계로 구분했는가? | Layer | 토스 |

→ 4개 언어를 모두 보여주는 앱 = "4축을 다 커버". 일부만 보여주면 해당 축만 부분 참조.

## 2. 골격 벤치마킹 조사 결과 (정성 / 1차)

### 2-1. Apple HIG — "축을 어떻게 나누는가"의 원전
- 구조 3단: **Components(종류로 묶음) / Layout·Hierarchy(레이어 위계) / Design Token(공유 값)**
  → 우리 Component축 / Layer축 / 토큰시스템과 동일 구조. 4축 설계의 정당성 확인.
- 핵심 원칙: 계층(중요한 것을 한눈에) · 일관성(익숙한 패턴 통일) · 명료성.
  → 우리 UI 이미지("깔끔·정돈·전 연령 가독성")와 정합.
- 색 규칙: **색은 단독으로 의미 전달 금지, 항상 라벨·형태와 함께.**
  → 우리 §원칙-7(상태=색 아닌 명암·테두리) 및 Status 태그 처리 방향과 일치.
- 접근성 수치 참고(2차 조사용): 본문 최소 17pt, 탭타깃 44x44pt, 8pt 그리드, 대비 4.5:1.

### 2-2. 토스(TDS) — "앱에 어떻게 구현되는가"의 실증
- TDS = 수백 컴포넌트가 개발과 연결되어 "하나의 언어"로 작동.
  → 우리 "주소(props) → 토큰 자동 적용" 철학과 동일.
- 단일 강조색 전략: 메인 컬러 #3182F7 하나로 메인버튼·대표색을 통일.
  → 우리도 단일 브랜드 중심 전략을 채택하되, Primary Brand는 Blue-Gray,
    Secondary Accent는 Indigo로 역할을 분리(2-3 참조).
- 7년 만의 색 시스템 개편: 타이포·스페이싱·상태를 토큰화 + Theme 시스템으로
  형태 변경 시 일괄 전파, 품질 유지.
  → 우리 "토큰 한 곳 변경 → 앱 전체 반영"의 업계 실증 사례.

### 2-3. 색상 방향 결론 [Blue-Gray 3층 확정 / 2606.13]
- **단일 브랜드 중심 전략 = 토스에서 차용** (단, Brand/Accent 역할 분리).
- **3층 구조 (서로 경쟁 아님, 역할 분담)**:
  - **1층 Brand = Blue-Gray** — 앱 분위기(헤더·포커스·브랜드·주요강조). "공간"을 칠함.
    신뢰·차분·깔끔·정돈·전연령. ("알바 계산기=전 연령 실용도구" 정체성에 최적)
  - **2층 Accent = Indigo** — 특별한 순간(Drawer헤더·시뮬레이터·특별카드). "여기 봐"를 칠함.
  - **3층 Semantic(불변) = Emerald(실수령)·Red(공제)·Amber(주의)** — "의미"를 칠함.
- **충돌 없음**: 세 층이 같은 화면 요소를 두고 다투지 않음.
  예) 실수령액 카드 = 배경/테두리는 Blue-Gray, 그 안 금액 숫자는 Emerald → 다른 요소에 적용.
  오히려 §원칙-1·2("한 의미=한 색")를 가장 잘 만족.
- ⚠ **조건 (BRAND-ACCENT-CONTRAST)**: 1층 Blue-Gray와 2층 Indigo가 둘 다 푸른 계열이라,
  토큰값 확정 시 **명도·채도 대비를 충분히 벌릴 것**. 안 그러면 강조(Indigo)와 기본(Blue-Gray)을
  사용자가 구별 못 함. 이 조건만 지키면 3층 구조 완전 안전.
- 선택상태(Status=Selected)는 **색 사용 금지** → 명암·테두리·폰트로만 (§원칙-7).

## 4. 벤치마킹 목적 재정의 — "4축 체계 검증" [핵심]

벤치마킹은 "예쁜 UI 찾기"가 아니라 **우리 4축 분류체계가 업계 실물에서 성립하는지 검증**하는 작업이다.
색·디테일이 아니라 *분류 방식*을 본다.

| 앱 | 검증 대상 | 무엇을 확인하나 |
|----|-----------|----------------|
| **1. Apple HIG** | Type·Component·Layer 축 | 이 3축이 정말 필요한가 (축의 존재 증명) — Feature 외 전부 다루는 원전 |
| **2. 토스** | Feature 축 + Layer 위계 | 기능 축이 실제로 존재하는가, 입력→결과→상세→BottomSheet→Toast 위계 구현 |
| **3. 카카오뱅크** | Emphasis 체계 | 10대 후반~60대 전 연령에 시선 위계가 어떻게 적용되나 (가독성은 토스보다 강함) |
| **4. 뱅크샐러드** | 숫자 중심 Result | 우리 ResultGrid와 가장 유사한 결과 강조 방식 |
| **5. 네이버 임금계산기** | 도메인 매핑 | 급여 기능 분해만 참조, 시각 체계는 반면교사 |

→ 추천 조사 순서: **Apple HIG → 토스 → 카카오뱅크 → 뱅크샐러드 → 네이버**.
→ 골격 단계에선 1·2번만 30분 가벼운 조사로 충분 (3~5는 토큰값 2차 조사 때 함께).

## 6. 검증 로그 — 4축 체계 실물 검증 (2606.13, 1차 정성)

**판정: 4축(Type×Component×Feature×Layer) 확정. 흔들림 없음. 05 재수정 불필요.**

### Apple HIG 대조 → Type·Component·Layer 존재성 ✅
- HIG 4레이어(Foundations·Patterns·Components·Technologies) = "철학→스펙" 위계. 우리 토큰계층(04)과 동형.
- **Component 축**: HIG Components = 버튼·탭바·슬라이더·알림 등 표준요소 상세스펙 → 우리 Component 12값의 원전.
- **Type 축 7값 ↔ HIG 대응물 (전부 존재)**:
  - Navigation ↔ Tab bar / Navigation
  - Overlay ↔ Presentation/Sheets
  - Feedback ↔ Notification
  - System ↔ Status bar
  - Display·Input·Action ↔ content / Inputs / menus and actions
  → Type은 발명이 아니라 업계 표준 재구성. 존재성 통과.
- **Layer 축**: HIG "visual hierarchy(주요 콘텐츠에 집중)" 원칙 = 우리 Input/Result/Detail 위계 근거.

### 토스 TDS 대조 → Feature·통일·Emphasis·위계 ✅
- **Component 통일**: TDS 수백 컴포넌트가 "하나의 언어"로 작동, 핵심 11개로 압축 → 우리 12종 통일과 동일 규모·접근.
- **Feature 축 실재**: "제품(기능)별 원칙 + 그 안의 컴포넌트 세트" 운용 = Feature 축과 동형.
- **Emphasis(위계)**: 큰 텍스트 시 아이콘도 같이 키워 위계 조정 → Primary/Secondary/Tertiary 사상과 일치.
- **주소→설계→상태 위계 동형**: TDS 가이드가 상위옵션(타입)→요소→상태변화(다크모드·2줄) 점층 구성 = 우리 주소→설계→Status 구조와 정확히 동일.
- **Semantic 분리**: 토스도 Semantic Color를 별도 체계 운용 → 우리 3층(Brand/Accent/Semantic) 분리와 일치.

### 추가 발견 (지금 미반영, 치환 단계 참고)
- 토스는 같은 Card를 primitive(원재료)·flat(조립본) 2층으로 둠. 공통 컴포넌트 실제 구현 시
  "기본 Card vs 조합 Card" 설계 참고. → Component 축 값에는 현재 반영 안 함, 메모만.

### 6-6. Apple HIG 검증에서 얻은 추가 통찰 (검증 결과 → 적용 원칙)
Apple 앱 벤치마킹의 핵심 가치는 시각 스타일이 아니라 **"제거의 기준"**에 있다.

1. **주인공은 하나만 둔다** — 애플은 화면마다 단 하나의 Primary만 남긴다 (계산기→숫자, Wallet→카드).
   나머지는 축소·숨김. → 본 프로젝트: Emphasis 적용 시 **Primary는 화면당 1개를 원칙**으로 한다.
2. **콘텐츠가 UI가 된다** — 장식보다 콘텐츠 우선, 콘텐츠 자체가 인터페이스.
   → 본 프로젝트: 장식성 카드 증가 지양 / 불필요한 아이콘 증가 지양 / 숫자·내용 중심 구성.
3. **위치와 행동의 일관성** — 동일 기능은 항상 동일 위치·행동 (Sheet 등장 방식, 뒤로가기 위치 고정).
   → 본 프로젝트: Component·Layer의 위치 규칙을 임의 변경하지 않는다.
4. **여백은 구조다** — 여백은 남는 공간이 아니라 위계 도구. 애플은 구분선보다 간격으로 그룹을 나눔.
   → 본 프로젝트: Divider 추가보다 Spacing 조정을 우선 검토한다.
5. **추가보다 제거를 우선한다** — 새 UI 추가 전, 기존 UI 제거 / 중복 정보 제거 / 숨김 처리를 먼저 검토.
   → **본 원칙은 UI-Audit(05) §X-5 '제거 우선 원칙'으로 정식 채택한다.** (상세 절차는 05 §X-6)

## 7. 검증 후 확정 원칙 (2606.13)

### 7-1. 벤치마킹 목적 재정의 (확정)
벤치마킹의 목적은 시각 스타일 모방이 아니라 다음 검증이다:
분류체계 / 위계 / 컴포넌트 구조 / 상태 관리 방식.
→ 색상·폰트·간격은 2차 검토(토큰값 단계) 대상.

### 7-2. Primitive → Semantic 구조 (향후 치환 참고)
공통 컴포넌트 구축 시 2단계 구조를 고려한다.
- **Primitive(원재료)**: Card · Button · Input
- **Semantic(의미 조립형)**: ResultCard · NetPayCard · AnnualLeaveCard
→ Primitive는 형태, Semantic은 프로젝트 의미를 담은 조립형. 코드 치환 단계 참고사항.
(Toss가 같은 Card를 primitive/flat 2층으로 둔 사례에서 도출.)

### 7-3. Type 실물 대응 기록 (Type 축 유지 근거)
Type 값은 추상 개념이 아니라 실제 디자인 시스템 대응물을 가진다.
- Navigation ↔ Tab Bar / Overlay ↔ Sheet / Feedback ↔ Notification / System ↔ Status Bar
→ Type 축은 유지한다.
