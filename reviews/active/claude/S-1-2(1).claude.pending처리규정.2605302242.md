## 논의 배경

C안(UI 비노출) 적용 시 pending 상태값을 어떻게 처리할 것인가에 대한 논의.

---

## pending 존재 이유 (기존)

```
pending = 병합주 여부에 따라 나중에 결정되는 상태
isFirst && !isMergedFirst && !week.isFullWeek ? "pending"
isLast && !week.isFullWeek ? "pending"
```

병합주/이전기간 판단이 있을 때 첫주/마지막주에 발생.

---

## S-1-2 설계 변경 후 pending 위상

### 설계 의도
- 병합주 개념 제거
- 이전기간 개념 제거
- 첫주/마지막주/일반주 전부 동일 처리
- 이용자 토글링 하나로 주휴 여부 결정

### 논리적 결론
pending의 존재 이유는 상실됨.

### 그러나 결정된 것
계산 로직은 건드리지 않는다 (C안 원칙).
따라서 pending 상태값은 코드 내부에 유지.

---

## 확정 기준 (내외부 규정)

### 내부 (코드)
```
pending 상태값 → 유지
pending 타입 정의 → 유지
pending 생성 로직 → 유지 (계산 로직 무변경 원칙)
```

### 외부 (UI)
```
pending → "주휴 발생 가능" 문구 노출 → 금지
pending → UI에서 발생/미발생 중 하나로 매핑하여 표시
```

### 처리 원칙
```
계산 로직 변경 없이 UI 표시만 변경 가능한지 확인 필요.
불가능하면 즉시 중단하고 보고.
```

---

## 하단 결과 문구 최종 확정

```
✔ 주휴 발생 (개근·주간 N시간)
✗ 주휴 미발생 (15시간 미달·주간 N시간)
✗ 주휴 미발생 (결근)
```

pending은 위 3가지 중 하나로 UI 표시.
"주휴 발생 가능" 문구는 개선된 UI에서 절대 사용 금지.
