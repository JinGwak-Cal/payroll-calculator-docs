# Premium 재설계 (03-0. 시간단위 정책 확정)

---

## 배경

Phase 1 검증 과정에서 발견된 구조적 문제:

- 연장수당: 총량 방식 (입력 N시간 = N시간 그대로 계산)
- 야간수당: 하루 기준 방식 (nightHoursPerDay × 일수 = 총 야간시간)
- 휴일수당: 총량 → 일수로 나눠 재변환 방식

세 수당의 시간 단위가 모두 다름.
칩 조합 입력에서 A state로 전달 시 변환 오류 가능성 존재.

---

## 확정 정책

**연장/야간/휴일/맞춤 모두 총량 시간 입력으로 통일한다.**

| 수당 | 기존 방식 | 변경 후 |
|------|-----------|---------|
| 연장 | 총량 (Phase 1에서 구현) | 총량 유지 ✅ |
| 야간 | 하루 기준 × 일수 | 총량으로 변경 |
| 휴일 | 총량 → /일수 변환 | 총량으로 통일 |
| 맞춤 | - | 총량 기준 |

---

## 구현 방향

### 야간수당
- `nightHoursManual` (총량) 신규 추가
- 기존 `nightHoursPerDay × 일수` 자동산출 방식 분리
- 칩 입력에서는 `nightHoursManual`만 사용

### 휴일수당
- 현재 총량 → /일수 변환 로직 제거
- `holidayHoursManual` (총량) 직접 전달로 통일

### 맞춤가산
- 총량 기준으로 입력받아 A state 전달

---

## 적용 원칙

```
사용자 입력: N시간 (총량)
A state 전달: N시간 그대로
calc-engine 계산: N × 시급 × 가산율
```

복잡한 단위 변환 없음. 모든 수당 동일한 흐름.

---

## 주의

- 이 정책 적용은 계산 로직 변경을 수반한다.
- calc-engine / use-calc 수정 필요.
- 기존 ResultGrid 야간/휴일 입력 UI도 영향받을 수 있음.
- 구현 전 별도 승인 필요.
