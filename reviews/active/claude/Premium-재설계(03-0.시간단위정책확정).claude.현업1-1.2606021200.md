# Premium 재설계 (03-0. 시간단위 정책 확정)

---

## 1. 시간 단위 정책

- Premium 칩 입력 시간은 총량 기준으로 통일한다.
- 연장 / 야간 / 휴일 / 맞춤 모두 총량 시간으로 해석한다.
- 휴일 라운드트립(총량 ÷ mult → × mult)은 최종 제거 방향으로 둔다.
- 단, 실제 제거는 Premium 구조 검증 후 진행한다.

---

## 2. 수동 vs 자동산출 우선순위

**기본 원칙:**
- 수동값이 있으면 수동값 우선
- 수동값이 없으면 자동산출 사용
- 수동과 자동을 합산하지 않는다.

**수동값 존재 기준:**
- trim 후 빈 문자열이 아니면 수동값 존재로 본다.
- "0"도 수동값이다.
- null / undefined / "" / 공백 문자열은 수동값 없음으로 본다.

---

## 3. EDGE 케이스 정책

**1) 수동값 0 입력**
수동으로 간주한다. 자동 복귀 금지.

**2) 수동 2시간 / 자동 4시간**
수동 2시간 우선. 자동 4시간 무시.

**3) 수동 입력 후 삭제**
빈 값이 되면 자동 복귀.

**4) 시간대 자동 + Premium 야간 동시 존재**
Premium 근무내역 중 야간 칩이 선택된 내역이 1건이라도 있으면 자동 야간 산출은 무시한다.
해당 내역의 시간이 0이어도 수동 우선으로 본다.
자동값과 Premium 값을 합산하지 않는다.

**5) 구버전 History 로드**
버전 분리 + 구버전 호환을 우선한다.
구버전 값을 무조건 0으로 폴백하지 않는다.

**6) 이용자 혼란 방지**
수동 입력 사용 중에는 UI에 다음 취지의 안내를 제공한다.
"수동 입력 사용 중 · 자동 계산 비활성"

---

## 4. 야간 자동산출 정책

- 야간 자동산출은 폐기하지 않는다.
- 시간대 입력 자동산출 모듈로 분리한다.
- Premium 총량 입력 모듈과 역할을 분리한다.
- Premium 입력값이 없을 때만 자동산출을 사용할 수 있다.

---

## 5. 휴일 라운드트립

- 현재 휴일은 총량 입력처럼 보이나 내부적으로 총량 ÷ mult → × mult 라운드트립을 거친다.
- 현재 계산 결과에는 큰 문제는 없어 보인다.
- 다만 장기적으로는 연장식 총량 방식으로 정리하는 것이 좋다.
- 왜 라운드트립 구조가 생겼는지 실태조사한다.

---

## 6. 작업 순서

0) Premium-재설계(03-0.시간단위정책확정) 문서 반영 ✅

2) Premium 재설계 완료 및 구조 검증
3) 야간 자동산출 모듈 분리
4) 연장 / 야간 / 휴일 총량 방식 최종 통일
